# Product Delivery Workflow Skill

## 适用场景
当用户表达以下意图时使用：
1. “帮我做这个需求方案”
2. “先出PRD和技术方案”
3. “我要可直接给开发的设计稿”
4. “帮我把这套流程沉淀成文档”

## 目标
在一个连续流程中产出 4 类资产：
1. 需求 PRD
2. 交互稿（低保真线框 + 状态流）
3. 高保真模拟稿（静态 H5，可直接评审/开发）
4. 技术方案（架构、接口、状态机、测试计划）

并确保内容可直接用于开发排期与实现。

---

## 标准执行流程（必须按顺序）

### Phase 0 - 对齐与约束
1. 明确业务目标、用户动作、成功标准
2. 明确范围：v1 in-scope / out-of-scope
3. 明确交付格式：文档 + 原型文件路径

输出：一句话目标 + 范围边界

### Phase 1 - 需求 PRD
必须包含：
1. 背景与问题
2. 目标与指标
3. 用户故事
4. 范围定义
5. 关键流程（成功流/失败流/取消流）
6. 验收标准（DoD）

建议落盘路径：
- `docs/02-requirements/YYYY-MM-DD-<topic>-prd.md`

### Phase 2 - 交互稿（低保真）
必须包含：
1. 状态流转图（mermaid）
2. 关键状态线框（至少 5 个：输入、接管、执行中、失败、完成）
3. 关键文案规范
4. 评审清单

建议落盘路径：
- `docs/03-interaction/YYYY-MM-DD-<topic>-wireframes.md`

### Phase 3 - 高保真模拟稿（静态 H5）
必须包含：
1. 可视化高保真页面（移动端优先）
2. 可切换多状态
3. 开发映射标记（如 `data-ui`）
4. 关键交互按钮（开始执行、失败注入、重试、取消）
5. 事件日志可视化区域

建议落盘路径：
- `docs/03-interaction/prototypes/<topic>-hifi.html`
- `docs/03-interaction/prototypes/<topic>-hifi-sim.html`
- `docs/03-interaction/prototypes/<topic>-handoff.md`

### Phase 4 - 技术方案
必须包含：
1. 架构分层
2. MCP tools 清单（输入/输出）
3. 执行状态机（Task/Step）
4. 事件协议
5. 安全与幂等策略
6. 测试计划与里程碑

建议落盘路径：
- `docs/plans/YYYY-MM-DD-<topic>-technical-spec.md`

---

## 输出质量要求
1. 可执行：每个方案都能映射到前后端任务
2. 可追踪：文档内明确接口、状态、验收标准
3. 可评审：交互稿和高保真稿可直接过会
4. 可复用：命名规范统一，便于后续迭代

---

## 交付后必须给出的总结格式
1. 已生成资产清单（带绝对路径）
2. 当前完成度（PRD/交互/模拟/技术）
3. 建议下一步（按优先级 1/2/3）

---

## 快速触发口令（用户可直接说）
1. “按产品交付流程来一版”
2. “帮我走 PRD + 交互 + 模拟 + 技术方案”
3. “用 product-delivery-workflow 跑这个需求”

---

## 本项目默认约定
1. 优先输出中文文档
2. 交互稿先低保真，再高保真模拟稿
3. 高保真模拟稿默认采用静态 H5 交付
4. 需求高风险动作必须有确认机制与审计设计
